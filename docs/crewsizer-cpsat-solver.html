1. Vue d'ensemble

Le module de dimensionnement CP-SAT de CrewSizer répond à la question fondamentale :

« Quel est l'effectif navigant minimum nécessaire pour couvrir le programme commercial, en respectant toutes les contraintes FTL ? »

Le solver utilise Google OR-Tools CP-SAT (Constraint Programming with Boolean Satisfiability), un solveur d'optimisation combinatoire de classe industrielle. Il modélise le problème comme un programme linéaire en variables binaires, où chaque décision (affecter un navigant à un bloc un jour donné) est une variable 0/1.

Entrée

Programme de vols résolu jour par jour + effectif disponible + règles FTL

Traitement

Optimisation sous contraintes : minimiser le nombre de navigants utilisés

Sortie

Effectif minimum par rang, marge, jours critiques, contrainte mordante, planning

Le solver est exécuté deux fois : une fois pour les PNT (Personnel Navigant Technique — pilotes CDB/OPL) et une fois pour les PNC (Personnel Navigant Commercial — CC/PNC). Les résultats sont combinés dans un CombinedSizingResult.

2. Architecture et flux de données

RunSizingCommand (MediatR CQRS) ↓ Chargement BDD • Scénario + Période • Membres équipage (actifs) • Indisponibilités • Blocs + TypeAvion • Semaines types + Calendrier ↓ Résolution programme jour/jour (Calendrier → SemaineType → BlocVol) ↓ SizingRequest • DailyPrograms [BlockInfo...] • AvailableCrew [CrewMemberInfo...] • FtlRules ↓ ┌────────────────────────┐ │ OrToolsSizingSolver │ │ CP-SAT (OR-Tools) │ │ Minimize(sum(isUsed)) │ │ 14 contraintes FTL │ └────────────────────────┘ ↓ SizingResult • MinimumCrewByRank • MarginByRank • CriticalDays • BindingConstraint • Assignments

Le handler RunSizingHandler charge toutes les données depuis PostgreSQL via Entity Framework Core, résout le programme jour par jour (calendrier → semaine type → blocs), puis construit le SizingRequest. Chaque BlockInfo porte les besoins en équipage issus du TypeAvion associé au bloc.

4. Variables de décision

Le modèle CP-SAT repose sur trois familles de variables binaires :

assign[c, d, b] ∈ {0, 1}

Vaut 1 si le navigant c est affecté au bloc b le jour d.

Dimension : numCrew × numDays × numBlocs(jour)

works[c, d] ∈ {0, 1}

Vaut 1 si le navigant c travaille le jour d (affecté à au moins un bloc).

Dimension : numCrew × numDays

isUsed[c] ∈ {0, 1}

Vaut 1 si le navigant c est utilisé au moins une fois sur toute la période.

Dimension : numCrew

Les liens entre ces variables sont établis par les contraintes C2 et C3.

5. Fonction objectif

Minimiser    Σc=0N-1 isUsed[c]

Le solver cherche à minimiser le nombre total de navigants utilisés tout en satisfaisant toutes les contraintes FTL. Le résultat est l'effectif strictement minimum nécessaire pour couvrir le programme commercial. 

// Ligne 289 — OrToolsSizingSolver.cs
model.Minimize(LinearExpr.Sum(isUsed));

Si le solver trouve une solution optimale (Status = Optimal), l'effectif retourné est garanti minimal. Si le solver atteint le timeout avec une solution réalisable (Status = Feasible), la solution est valide mais pas nécessairement optimale.

6. Contraintes (C1–C14)

Toutes les contraintes sont dures (hard constraints) : le solver ne peut pas les violer. Si aucune solution ne respecte toutes les contraintes, le problème est déclaré Infeasible.

C1 Indisponibilités (congés, formation)
Si le navigant c est indisponible le jour d :   assign[c, d, b] = 0   ∀ b

Un navigant en congé, en formation ou en arrêt ne peut pas être affecté à un bloc ce jour-là. Les dates d'indisponibilité sont chargées depuis la BDD. 

if (crewByIndex[c].UnavailableDates.Contains(date))
    model.Add(assign[c][d][b] == 0);
C2 Maximum 1 bloc par navigant par jour
Σb assign[c, d, b] ≤ 1    ∀ c, d

Un navigant ne peut être affecté qu'à un seul bloc par jour. Cette contrainte lie également works[c,d] à assign[c,d,*] : works[c,d] = 1 ssi au moins un assign[c,d,b] = 1. 

model.Add(LinearExpr.Sum(blockVars) <= 1);
model.Add(LinearExpr.Sum(blockVars) >= 1).OnlyEnforceIf(works[c][d]);
model.Add(LinearExpr.Sum(blockVars) == 0).OnlyEnforceIf(works[c][d].Not());
C3 Lien isUsed ↔ works
isUsed[c] = max(works[c, 0], works[c, 1], ..., works[c, D-1])

Le navigant est marqué « utilisé » dès qu'il travaille au moins un jour. C'est cette variable qui est minimisée dans la fonction objectif. 

model.AddMaxEquality(isUsed[c], works[c]);
C4 Couverture des blocs — PNT (pilotes)
Σc ∈ CDB assign[c, d, b] = RequiredCdb    ∀ d, b
Σc ∈ OPL assign[c, d, b] = RequiredOpl    ∀ d, b

Chaque bloc doit être couvert par exactement le nombre requis de CDB et d'OPL. Ces valeurs proviennent du TypeAvion du bloc (ex: ATR 72-600 → 1 CDB + 1 OPL). L'égalité stricte (« == ») garantit qu'on n'affecte ni trop ni trop peu de pilotes. 

var cdbVars = crewByRank[CrewRank.CDB]
    .Select(c => assign[c][d][b]).ToArray();
model.Add(LinearExpr.Sum(cdbVars) == block.RequiredCdb);
C4a Couverture cabine — CC minimum
Σc ∈ CC assign[c, d, b] ≥ RequiredCc    ∀ d, b

Au moins RequiredCc chefs de cabine doivent être affectés à chaque bloc. Un PNC ne peut pas remplacer un CC (le CC a des responsabilités spécifiques).

C4b Couverture cabine — total (polyvalence CC→PNC)
Σc ∈ CC ∪ PNC assign[c, d, b] ≥ RequiredCc + RequiredPnc    ∀ d, b

Le nombre total de personnel cabine (CC + PNC) doit couvrir l'ensemble des besoins. Un CC peut occuper un poste PNC (polyvalence), mais pas l'inverse. Voir section 8 pour le détail. 

var totalCabinRequired = block.RequiredCc + block.RequiredPnc;
var allCabinVars = ccVars.Concat(pncVars).ToArray();
model.Add(LinearExpr.Sum(allCabinVars) >= totalCabinRequired);
C5 Temps de service cumulatif — 7 jours glissants ORO.FTL
Σd'=dd+6 Σb assign[c, d', b] × DutyMinutes(d', b) ≤ 3 600 min (60h)    ∀ c, d

Dans toute fenêtre glissante de 7 jours, le temps de service cumulé d'un navigant ne doit pas dépasser 60 heures.

C6 Temps de service cumulatif — 14 jours glissants ORO.FTL
Σd'=dd+13 Σb assign[c, d', b] × DutyMinutes(d', b) ≤ 6 600 min (110h)    ∀ c, d
C7 Temps de service cumulatif — 28 jours glissants ORO.FTL
Σd'=dd+27 Σb assign[c, d', b] × DutyMinutes(d', b) ≤ 11 400 min (190h)    ∀ c, d

Activé uniquement si l'horizon de planification couvre ≥ 28 jours.

C8 HDV cumulative — 28 jours glissants ORO.FTL
Σd'=dd+27 Σb assign[c, d', b] × BlockTimeMinutes(d', b) ≤ 6 000 min (100h)    ∀ c, d

Heures de vol (HDV) cumulées sur 28 jours glissants. Distinct du temps de service (DP) : le HDV ne compte que le temps en vol, pas la préparation ni le post-vol.

C9 Jours OFF minimum — 28 jours Délib. 77
Σd'=dd+27 works[c, d'] ≤ 20   (= 28 − 8)    ∀ c, d

Au moins 8 jours OFF par fenêtre glissante de 28 jours (soit maximum 20 jours travaillés).

C10 Repos minimum entre deux services ORO.FTL
Si repos(bloc b1 jour d, bloc b2 jour d+1) < 720 min (12h) :    assign[c, d, b1] + assign[c, d+1, b2] ≤ 1

Si le repos entre la fin d'un bloc et le début du bloc suivant est inférieur à 12 heures, le même navigant ne peut pas être affecté aux deux blocs consécutivement. 

int restMinutes = (24 * 60 - endToday) + startTomorrow;
if (restMinutes < minRestMinutes)
    model.Add(assign[c][d][b1] + assign[c][d + 1][b2] <= 1);
C11a Maximum 6 jours consécutifs ON Délib. 77
Σd'=dd+6 works[c, d'] ≤ 6    ∀ c, d

Dans toute fenêtre de 7 jours consécutifs, il y a au moins 1 jour OFF. Autrement dit : jamais plus de 6 jours travaillés d'affilée.

C11b Pas de jour OFF isolé (ON-OFF-ON interdit) Convention
works[c, d−1] − works[c, d] + works[c, d+1] ≤ 1    ∀ c, d

Le pattern ON-OFF-ON est interdit. Quand un navigant s'arrête, il doit s'arrêter au moins 2 jours consécutifs. Un seul jour OFF entre deux jours ON ne constitue pas un repos valide.

PatternCalculValide ?
ON–OFF–ON1 − 0 + 1 = 2 > 1Interdit
ON–OFF–OFF1 − 0 + 0 = 1 ≤ 1OK
ON–ON–ON1 − 1 + 1 = 1 ≤ 1OK
OFF–OFF–ON0 − 0 + 1 = 1 ≤ 1OK
C12 Repos hebdomadaire — 36h + 2 nuitées / semaine civile Délib. 77
Par semaine civile (lundi→dimanche) : au moins 1 séquence de 2 jours consécutifs OFF

Chaque semaine civile ISO (lundi au dimanche), le navigant doit bénéficier d'un repos de 36 heures incluant 2 nuitées locales consécutives. En modélisation journalière : 2 jours consécutifs OFF dans la semaine.

Distinction avec C11 : C11 est une fenêtre glissante, C12 s'applique à la semaine civile fixe.

C13 Repos mensuel week-end — 3 jours incluant Sam+Dim Convention
Par mois calendaire : au moins 1 repos de 3 jours consécutifs incluant Samedi ET Dimanche

Une fois par mois, chaque navigant doit avoir un repos de 3 jours consécutifs incluant obligatoirement un samedi et un dimanche. Les seules combinaisons possibles sont :

  • Vendredi + Samedi + Dimanche
  • Samedi + Dimanche + Lundi

Le solver crée des variables booléennes pour chaque week-end du mois (option VSD et option SDL), puis impose qu'au moins une de ces variables soit vraie.

C14 RDOV — 2,5 jours de bureau / semaine civile Convention

Le navigant affecté au rôle RDOV effectue 2,5 jours de bureau par semaine civile. Il n'est disponible pour les vols que 2,5 jours par semaine et ne vole jamais le samedi ni le dimanche.

Cette contrainte est modélisée en 3 sous-contraintes :

C14a — Repos week-end obligatoire

works[c, d] = 0    si d ∈ {Samedi, Dimanche}    ∀ c ∈ RDOV

C14b — Max 3 jours de vol par semaine civile

Σd ∈ Lun..Ven(semaine) works[c, d] ≤ 3    ∀ c ∈ RDOV, ∀ semaine civile

Plafond entier de 2,5 : permet jusqu'à 3 jours de vol dans une semaine isolée.

C14c — Max 5 jours de vol par 2 semaines consécutives

Σd ∈ jours ouvrés(sem. N + sem. N+1) works[c, d] ≤ 5    ∀ c ∈ RDOV

Garantit la moyenne exacte de 2,5 jours/semaine. Répartitions possibles : 3+2, 2+3 ou 2+2 (sous-utilisation). 

// AddRdovConstraint — OrToolsSizingSolver.cs
if (dow is DayOfWeek.Saturday or DayOfWeek.Sunday)
    model.Add(works[c][d] == 0);           // C14a

model.Add(LinearExpr.Sum(weekVars) <= 3);  // C14b

model.Add(LinearExpr.Sum(twoWeekVars) <= 5); // C14c

7. Dictionnaires de contraintes par source réglementaire

Les 14 contraintes du solver sont classées en 4 catégories selon leur source juridique. Cette classification permet de tracer l'origine de chaque règle et d'identifier quelle réglementation est la plus limitative pour le dimensionnement.

Le système utilise un ConstraintRegistry qui fusionne les 3 sources paramétriques en prenant la valeur la plus restrictive pour chaque paramètre : pour les maximums (duty, HDV, jours consécutifs), c'est la plus basse ; pour les minimums (repos, jours OFF), c'est la plus haute.

Elles s'appliquent à tout exploitant aérien soumis au règlement EASA ou à ses transpositions nationales (arrêté du 28 juin 2011 pour les territoires hors EASA). Elles fixent les plafonds cumulatifs de temps de service et d'heures de vol, ainsi que le repos minimum inter-services.

  • C11b — interdit les jours OFF isolés (pattern ON-OFF-ON), garantissant des repos de qualité d'au moins 2 jours consécutifs.
  • C13 — impose un repos week-end mensuel de 3 jours consécutifs incluant samedi + dimanche, assurant un équilibre vie professionnelle / vie privée.
  • C14 — navigant RDOV : repos obligatoire samedi + dimanche, au plus 2,5 jours de vol par semaine civile (modélisé en 3 sous-contraintes C14a/b/c).

7.5 Fusion multi-sources — ConstraintRegistry.MergeToFtlRules()

Lorsque plusieurs sources définissent le même paramètre, le registre applique la règle de la valeur la plus restrictive :

Type de paramètreRègle de fusionExemple
Maximum (duty, HDV, jours consécutifs) Math.Min() — la plus basse gagne Si ORO.FTL dit 7j consécutifs max et Délib. 77 dit 6 → 6
Minimum (repos, jours OFF) Math.Max() — la plus haute gagne Si ORO.FTL dit 10h de repos et Convention dit 12h → 12h 
// ConstraintRegistry.cs — Fusion multi-sources
int Resolve(string propertyName, int fallback)
{
    var matching = all.Where(c => c.FtlRulesProperty == propertyName);
    if (first.LowerIsMoreRestrictive == true)
        return matching.Min(c => c.DefaultValue);  // max duty → min gagne
    if (first.LowerIsMoreRestrictive == false)
        return matching.Max(c => c.DefaultValue);  // min repos → max gagne
}

8. Polyvalence CC → PNC

Règle métier : un Chef de Cabine (CC) peut remplir la fonction de PNC sur un bloc, mais un PNC ne peut pas remplacer un CC. Le CC possède des compétences et responsabilités supplémentaires (sécurité, gestion cabine).

Cette polyvalence est modélisée par la combinaison des contraintes C4a et C4b :

ContrainteFormuleSémantique
C4a Σ(CC) ≥ RequiredCc Au moins N CC — un PNC ne peut pas être CC
C4b Σ(CC + PNC) ≥ RequiredCc + RequiredPnc Total cabine suffisant — un CC peut remplir un poste PNC

Exemples

ATR 72-600 (Air Calédonie)

RequiredCc = 1, RequiredPnc = 0

C4a : ≥ 1 CC → 1 CC suffit

C4b : ≥ 1 total → redondante avec C4a

Avion fictif (CC=1, PNC=1)

RequiredCc = 1, RequiredPnc = 1

C4a : ≥ 1 CC

C4b : ≥ 2 total cabine

Solutions : 1 CC + 1 PNC ou 2 CC

Le résultat final combine les deux passes (PNT + PNC) : 

public sealed record CombinedSizingResult
{
    public required SizingResult PntResult { get; init; }
    public required SizingResult PncResult { get; init; }
    public bool IsBothFeasible => PntResult.IsFeasible && PncResult.IsFeasible;
}

10. Analyse de la contrainte mordante

Après résolution, le solver analyse la solution pour identifier la contrainte FTL la plus serrée (« binding constraint ») — celle qui détermine le plus l'effectif minimum.

La méthode DetermineBindingConstraint() calcule 8 ratios d'utilisation (0% à 100%) pour chaque navigant, puis retourne le maximum global :

CodeRatioSourceFormuleLimite
C5TSV 7jORO.FTLmax duty 7j réel / 60h60h
C6TSV 14jORO.FTLmax duty 14j réel / 110h110h
C8HDV 28jORO.FTLmax HDV 28j réel / 100h100h
C9Jours ON 28jDélib. 77max jours travaillés / 2020 (= 28 − 8 OFF)
C11aJours consécutifsDélib. 77max consécutifs / 66 jours
C12Repos hebdoDélib. 77jours travaillés / max autorisé dans la semaine5/7
C13Repos WE mensuelConventionWE travaillés / (total WE − 1)n−1 WE
C14RDOV 2,5j/sem.Conventionmax jours volés RDOV / 2,5 par semaine civile2,5 j/sem

Le résultat inclut désormais le code et la source réglementaire de la contrainte mordante, via les champs BindingConstraintCode et BindingConstraintSource du SizingResult.

Exemple de sortie : "TSV 7j (60h max) — utilisation 98%" avec code C5 et source ORO.FTL signifie que la limite ORO.FTL.210(b) de 60h sur 7 jours est la contrainte la plus limitative, avec un navigant atteignant 98% de la limite.
